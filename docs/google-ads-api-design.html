1. Tool Overview

Painel DDG Tráfego Pago is a proprietary internal web dashboard developed by Dose de Growth, a Brazilian digital marketing agency, to consolidate paid-traffic operations across our client portfolio. The tool unifies Google Ads and Meta Ads data into a single operational interface used by our in-house traffic managers.

Our agency currently manages approximately 15–20 active Google Ads accounts under MCC 513-194-3362, spanning e-commerce, healthcare clinics, beauty/wellness services, real estate, education, and B2B services.

The Google Ads API is required because our manual workflow today (logging in to each client account separately, exporting CSVs, copying numbers into spreadsheets, building custom client reports) is unsustainable as our portfolio grows. The Painel DDG replaces this manual process with a unified, automated read-and-analyze layer on top of our agency's day-to-day operations.

Key principle: the Painel DDG is not a public SaaS, is not resold, is not a third-party reporting product, and is not made available to anyone outside Dose de Growth's payroll and contracted account managers. It is strictly an internal operational tool for the same agency that already manages these client accounts under its MCC.

1.1 Primary purpose

  1. Read-only data consolidation: pull campaign, ad group, ad, keyword, and search-term performance data from client accounts our agency already manages.
  2. Cross-platform unified reporting: display Google Ads + Meta Ads side-by-side with normalized metrics (spend, impressions, clicks, CTR, CPC, conversions, CPA, revenue, ROAS).
  3. Anomaly detection & AI-assisted analysis: highlight spend spikes, ROAS drops, and unusual CPA patterns vs. 7d/14d baselines, with explanatory narratives generated by the Anthropic Claude API.
  4. Search-term auditing: surface high-spend / zero-conversion search terms to enable bulk negative keyword additions through our interface.
  5. Offline conversion uploads (Enhanced Conversions for Leads): forward real CRM closures (deals closed by our sales team) back to Google Ads so Smart Bidding optimizes toward customers who actually purchase, not just form-fillers.

3. Target Users & Access Model

The Painel DDG is an internal-use tool. The following user roles are defined:

RoleWhoAccess Level
adm_geral DDG founder / agency operations lead (currently 1 person) Full access to all client data, configuration, integrations, and offline-conversion uploads
gestor_ddg DDG traffic managers (W2 employees and long-term contractors) Read/write access scoped to the clients explicitly assigned to them
viewer_cliente (Phase 5+) Direct contacts at clients we already manage Read-only, restricted to their own brand's data, white-labeled view

For the API application under review, we are explicitly requesting internal users only. The viewer_cliente role exposes aggregated reporting only (no API write capabilities), and we will revisit the API access scope before exposing this read-only public-facing layer in the future.

Authentication is handled by Supabase Auth (email + password and Google magic link). Row Level Security (RLS) is enforced at the database level so each user can only query data belonging to clients they are explicitly linked to.

6. Authentication & Authorization Flow

OAuth 2.0 is implemented exactly as specified by the Google Ads API authorization guide. The agency user (DDG admin) authenticates with their Google account that has access to the MCC; the resulting refresh token is encrypted at rest and used for all subsequent API calls on behalf of that MCC.

7. Data Storage & Sync Strategy

7.1 Snapshot model

Performance data is stored as immutable daily snapshots. Each sync inserts new rows; we never update existing rows. This guarantees an auditable historical trail and makes reconciliation with the Google Ads UI deterministic.

Key tables (under trafego_ddg schema):

  • campanhas_snapshot — one row per (cliente, plataforma, campanha_id, data)
  • adsets_snapshot — same pattern, ad-group level
  • ads_snapshot — ad-level metrics + creative thumbnail URLs
  • keywords_snapshot — Search keywords with quality score
  • search_terms — search terms triggering ads (with status flag for negative-keyword workflow)

8. Use Cases & User Stories

Use Case 1 — Daily morning briefing

Maria, a DDG traffic manager, opens the Painel at 9 AM and immediately sees, for each of her assigned clients: yesterday's spend vs. budget, conversions vs. 7-day baseline, and a sorted list of anomalies (e.g. "Petderma — Search ROAS dropped 38% vs. 7d baseline" with an AI-generated explanation pointing to a suspect ad group). Without the API, this takes 45–60 minutes of manual logins; with the API, it is a single page that loads instantly from our snapshot tables.

Use Case 2 — Search-term audit and bulk negation

Once a week, Maria filters the search_terms table for terms with cost > R$200 and zero conversions in the last 30 days. She reviews the list, multi-selects irrelevant terms, and clicks "Add as negative keyword". The Painel issues a MutateAdGroupCriteria request for each selection. Every action is logged in logs with user_id, timestamp, payload, and API response — providing a complete audit trail.

Use Case 3 — Real fechamentos feeding Smart Bidding

When a client closes a deal in DDG's internal commercial CRM, a webhook fires to the Painel containing email, phone, gclid, and deal value. The push-offline-conversions job batches these events and uploads them via OfflineUserDataJobService as Enhanced Conversions for Leads. Within 24 hours the conversion appears in Google Ads, and Smart Bidding starts optimizing for customers who actually purchase, not just form-fillers — substantially improving ROAS over the next 2–4 weeks of campaign learning.

9. Rate Limiting & Error Handling

9.1 Rate limiting

  • Application-level throttle: hard cap of 10 concurrent API calls across all sync jobs
  • Per-account sequential processing (no parallel calls within the same customer-id)
  • Exponential backoff on retryable errors: 1s, 2s, 4s, 8s, 16s (max 5 retries)
  • Daily call counter persisted in logs for monitoring

9.2 Error handling

Error typeHandling
UNAUTHENTICATEDRefresh access token; if refresh token revoked, mark account status_google = 'erro' and notify admin to re-authorize
QUOTA_ERRORBackoff 60 minutes; resume on next cron tick
RESOURCE_EXHAUSTEDExponential backoff up to 5 retries; alert admin if exceeded
INVALID_ARGUMENT (GAQL syntax)Log full payload, mark sync as failed for that resource only, continue with others
Generic 5xxStandard retry with backoff

Every API call (success or failure) is recorded in the logs table for retroactive audit and quota tracking.

10. Security, Privacy & Compliance

  • OAuth tokens encrypted at rest via Supabase Vault (AES-256-GCM)
  • Row Level Security on every table — users can only access data linked to clients in their clientes_users mapping
  • HTTPS only — TLS 1.3 enforced by Vercel and Supabase
  • PII hashing for offline conversions — emails and phones SHA-256-hashed before transmission to Google Ads (also a Google policy requirement for Enhanced Conversions)
  • LGPD compliance (Brazilian GDPR equivalent): clients can request data export and deletion; data processing agreements in place with all clients
  • Audit log: every API mutation (negative keyword adds, offline uploads) is logged with user_id, timestamp, and payload
  • No data resale: Google Ads API data is never exposed outside DDG operations or shared with third parties
  • Daily backups: Supabase automatic Point-in-Time Recovery enabled

12. Compliance with Google Ads API Policies

We acknowledge and will adhere to all Google Ads API policies, including:

  • The Required Minimum Functionality requirements for any tools accessing the API
  • The Google Ads API Terms and Conditions
  • The Google Ads policies regarding offline conversion uploads, hashed user data, and data retention
  • Keeping the API contact email up-to-date and responding to all inquiries from the API team within 5 business days
  • Not reselling or sublicensing API access
  • Not using the API to create automated campaigns or bidding strategies that operate outside human review

All UI mutations (negative keywords, offline uploads) require explicit user action and are logged. No campaigns or bid changes are made automatically without human approval.
